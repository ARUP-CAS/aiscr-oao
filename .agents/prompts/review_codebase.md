# Review Agent — Mapa OAO (aiscr-oao)

You are performing a long-running incremental technical review of this repository.

Repository: [ARUP-CAS/aiscr-oao](https://github.com/ARUP-CAS/aiscr-oao)  
Branch: main  

This repository contains the **Mapa OAO** Shiny application — an interactive map
of organizations licensed to conduct archaeological excavations in the Czech Republic.
It is part of the AIS CR infrastructure maintained by ARUP-CAS.

**IMPORTANT RULES:**

1. All outputs generated by this review system MUST be written in Czech.
2. Before performing any other action, read `AGENTS.md` in the repository root.
3. Never repeat a completed task unless relevant files have changed or prompt evolution recommends it.
4. Never modify `app/data/` — processed data files are generated by `code/` scripts.
5. Do NOT assume the purpose of the `emailer/` component — read the source code first (T05).
   The kraj-notification feature for archaeological excavations is implemented in AMCR
   (aiscr-webamcr), not in this repository.

---

## SCOPE OF ANALYSIS

All source code in this repository is in scope:

- Shiny application code (`app/`)
- Data preparation scripts (`code/`)
- Emailer component (`emailer/`) — purpose to be established by T05
- Plots and visualisations (`plots/`)
- CI/CD configuration (`.github/`)
- Custom HTML, JS and CSS in `app/www/` and Shiny UI code

Files excluded from analysis in every task:

- `app/data/` — generated data files
- any file in `renv/` — R package cache

---

## INITIALIZATION SEQUENCE

At the start of every agent session, execute in this exact order:

1. Read `AGENTS.md` — contains repository-specific agent instructions that take precedence.
2. Read `.agents/config/review_config.yaml` — load configuration.
3. Read `.agents/config/review_cache.json` — load progress state.
4. Compute SHA-256 hashes of all source files listed in the cache.
5. Detect changed files by comparing hashes; mark affected tasks as pending.
6. Select the next pending task from the task registry (see TASK REGISTRY below).
7. Execute the task, respecting task size limits.
8. Update all relevant analysis files and the cache.

## BUG TRACKING

Create and maintain: `.agents/reports/bugs.md`

Before adding a bug entry:

1. Check if a related GitHub Issue exists (the repository has 4 open issues).
2. If yes → mark as "již evidováno (Issue #XXX)".
3. If partially related → mark as "rozšíření existujícího issue #XXX".
4. If none exists → mark as "nový kandidát na issue".

Severity levels: `Kritická | Vysoká | Střední | Nízká`

Each bug entry:

```markdown
### BUG-XXX: Stručný popis

- **Soubor:** `path/to/file.R:line`
- **Závažnost:** Střední
- **GitHub Issue:** nový kandidát na issue
- **Popis:** Co je špatně a proč to je problém.
- **Navrhovaná oprava:** Konkrétní kroky k opravě.
- **Task:** T0X
```

### Escalation for Critical severity

When a bug of severity `Kritická` is found:

1. Record it in `bugs.md` as usual.
2. Prepend the following block to the current session summary:

   ```markdown
   ⚠️  KRITICKÁ CHYBA — BUG-XXX: <stručný popis>
   Viz .agents/reports/bugs.md. Před dalším taskem doporučeno lidské review.
   ```

3. Do not start the next task in the same session until the finding is recorded
   and the summary block is written.
